## 当前已经成型的模块

### 1. 桌面软件主链路

- Electron 正式版主链路已稳定：
  - `Renderer -> Preload -> IPC -> Main -> AppService -> Service/Repo -> SQLite`
- `xplorone.db` 已成为运行时唯一正式数据库，预算表现已并入主库
- 默认运行时目录已切到 `%APPDATA%\\XplorOne`
- 旧 `%APPDATA%\\finance-v1` 已有复制迁移与旧 DB 文件名兼容补齐；旧 `xplorone-budget.db / budget.db` 会在启动期自动迁入主库
- 安装身份、环境变量主族和凭据 namespace 已切到 `XplorOne / XPLORONE_* / xplorone-ai`
- Web 预览与正式 Electron 的环境边界已经明确

### 2. 多账本体系

- 多账本底层已打通
- 当前账本切换已接通
- 账本删除与归档已拆开：
  - `归档`：保留记录
  - `删除`：二次确认后真实删除
- 已归档账本有独立入口，并支持取消归档
- 已归档账本当前已支持导出为单个 `.xpl` 迁移包，并支持从 `.xpl` 导入为新的活动账本
- 默认账本 seed 不会再把已删账本按名字补回来

### 3. 模板化新建账本

- 模板只用于“新建账本”
- 不支持给已有账本重新套模板
- 当前可用模板：
  - `personal_v1`
  - `family_v1`
  - `business_v1`
- `create_book` 已按单事务实现：
  - 创建 book
  - 写入 `template_code`
  - 创建收入/支出根分类
  - 复制模板分类树
  - 复制模板账户
- 任一步失败整笔回滚

## 当前已经可用的 AI 对话系统

### 1. 对话模块结构

当前对话系统已稳定为 4 个模块：

- `查询`
- `录入`
- `分析`
- `自由对话`

这 4 个模块在页面上已经分开，不再混成一个模糊聊天框。

### 2. AI 查询

- `chat_query` 已形成稳定主链
- 当前走的是：
  - 本地规则优先
  - AI 兜底解析
  - 再执行标准查询
- 当前已稳定支持的方向包括：
  - 汇总查询
  - 账户余额
  - 账户流水
  - 流水笔数
  - 分类汇总
  - 简单期间对比
  - 资产 / 负债 / 净资产 / 净收入等常见口径
  - Top N / 排名查询

### 3. AI 录入

- `chat_entry` 已接通
- 查询和录入从新对话开始已经拆开
- 当前录入链路已具备实际可用性：
  - 单笔录入
  - 多笔录入
  - 草稿整理
  - 补充信息
  - 用户确认
  - 跳转记一笔 / 记多笔
  - 直接提交
- 录入成功后会回写聊天结果，形成闭环

### 4. AI 分析

- `chat_analysis` 已接通
- 分析模式当前已具备：
  - 财务顾问角色
  - 本地数据上下文装配
  - readiness 判断
  - 分析回答生成
- 简单问题会路由回查询链路

### 5. 自由对话

- `chat_free_talk` 已接通
- 当前定位明确：
  - 开放式财务助手
  - 允许投资、税务、泛经营问题
- 当前已支持个性化摘要注入（前提是开关开启）

### 6. 本地对外 API / 外部只读查询基础

- 当前项目已经具备本地 HTTP API，不只依赖 Electron IPC
- API 默认使用 Bearer token 校验
- 当前默认监听本机回环地址
- 已具备：
  - `/health`
  - `/api/runtime`
  - `/api/execute`
  - `/api/query/execute`
- 其中 `/api/query/execute` 已经具备：
  - `request_id`
  - `source`
  - `scope`
  - 审计日志
  这些外部只读查询所需的基础字段
- 当前代码中已经出现：
  - `workbuddy`
  - `workbuddy:read`
  这说明“外部 agent 只读查询”这条路在系统里已经有明确语义基础

### 7. 本地 MCP 只读查询已落地

- 当前项目已经额外落地了本地 MCP 服务，主要定位是把现有只读查询能力标准化暴露给外部 agent。
- 当前 MCP 已固定为 **只读查询 v1**：
  - 不做写入
  - 不做切账本
  - 不做跨账本显式查询
- 当前 MCP 已稳定暴露 6 个工具：
  - `get_runtime_info`
  - `list_books`
  - `list_reference_data`
  - `query_transactions`
  - `query_account_transactions`
  - `get_finance_summary`
- 当前 MCP 主链路已经不是纸面设计，而是已实际打通：
  - `MCP -> 本地 HTTP API -> QueryRequestService / QueryExecutor -> SQLite`
- 当前 Codex 桌面端真实入口已经实测通过：
  - 能发现 tools
  - 能实际调用 `get_runtime_info / list_books / query_transactions`
  - 说明外部 agent 只读接入已经进入“可用”状态，而不是纯开发草图
- 当前 MCP 也已经不再停留在“开发脚本模式”：
  - 主程序会后台托管 MCP 生命周期
  - 设置页已有 `XplorOne MCP` 状态区
  - 设置页可分别管理 `Codex / WorkBuddy` 两类客户端 token
- 当前 MCP client token 已切到 Electron `safeStorage` 加密存储：
  - 设置页默认只显示脱敏预览
  - 支持复制 token
  - 支持短时显示完整 token
  - 支持重新生成 token
- 当前 Local API 与 MCP 的依赖关系也已收口：
  - MCP 可按需临时拉起 Local API
  - MCP 启动失败时，会回收这次临时拉起的 API
  - MCP 关闭时，不会误关用户手动开启的 Local API

## 当前已经明确的运行边界

### 1. Web 预览和正式软件不是同一能力面

- 正式 Electron：
  - 可调系统目录/文件选择器
  - 可直接打开本地路径
  - 可直接调用主进程截图能力导出页面局部图片
- Web 预览：
  - 不能真正拉起系统目录/文件选择器
  - 只能 fallback
  - 复杂 DOM / SVG 页面导出时仍可能受浏览器 canvas 安全限制影响

### 2. 数据主键不是业务序号

- `accounts.id`、`categories.id` 是全表自增主键
- 不是按 `book_id` 各自从 1 开始
- 模板创建、恢复、删除后重建会让 id 跳大
- 这是正常现象，不代表表损坏

### 3. 个性化设置当前按单用户 Profile 理解

- 当前所有个性化设置均视为当前 `baseDir` 下唯一活跃 Profile 的配置
- 还没有真正的多 Profile / 多用户切换层

### 4. 外部 agent 集成当前边界

- 当前更适合优先开放 **只读查询**
- 查询结果可以直接返回给外部 agent，由 agent 自己的模型负责解释和组织回答
- 软件内置模型当前仍主要服务于软件内部：
  - `chat_query`
  - `chat_analysis`
  - `chat_free_talk`
- 也就是说：
  - 软件外部集成优先复用本地 API 的结构化查询能力
  - 软件内部 AI 继续负责软件内对话与分析体验
