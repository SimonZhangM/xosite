适用范围：
- 本文件用于“每个新线程开始编程前先读一遍”。
- 重点记录的是开发过程中的硬经验，不是产品介绍。

建议新线程开场固定动作：
1. 先读 [D:\APP\finance-v1\SYSTEM_ARCHITECTURE.md](/D:/APP/finance-v1/SYSTEM_ARCHITECTURE.md)
2. 再读本文件
3. 再读 [D:\APP\finance-v1\PROJECT_MEMORY.md](/D:/APP/finance-v1/PROJECT_MEMORY.md)
4. 再看 `git status`

## 1. 优先改最小作用域

这是当前项目最重要的经验之一。

- 修局部问题时，不要轻易把改动上提到全局结构。
- 尤其不要为了修某个子面板，顺手去大改：
  - `src/renderer/App.tsx`
  - `src/renderer/pages/SettingsPage.tsx`
- 最近已经踩过坑：本来只是修 `DataSettingsPanel`，后来把改动扩到 `App.tsx / SettingsPage.tsx`，结果把局部问题放大成整页甚至整站异常。

优先原则：
- 先改出问题的具体组件
- 再改该组件直接依赖的样式
- 只有确认根因在更上层时，才往上提改动范围

## 2. 大文件只做手术刀式修改

这些文件改动风险高：
- `src/renderer/App.tsx`
- `src/renderer/pages/SettingsPage.tsx`
- `src/renderer/pages/HomeLedgerPage.tsx`
- 以及其他逻辑密集的大文件

规则：
- 先明确“只改哪一小块”
- 只做最小补丁
- 不做整文件重写
- 每改完一轮就及时 `build`

## 3. Web 预览和正式 Electron 是两种环境

开发时一定先区分当前运行环境。

### Web 预览
- 不能真正调用系统目录/文件选择器
- 不能真正直接拉起本地文件夹窗口
- 需要 fallback：
  - 手动输入路径
  - 复制路径后手动打开
- 如需在 Web 预览 / 纯 Node 调试态验证 AI 链路，当前只能显式开启：
  - `XPLORONE_NODE_AI_COMPAT=1`
  - 或兼容旧变量 `FINANCE_NODE_AI_COMPAT=1`
- 这条只用于预览 / 调试，不代表 Node 运行时重新成为正式 AI 入口

### 正式 Electron
- 可以调用系统目录/文件选择器
- 可以直接打开本地目录

结论：
- 在 Web 预览里看到“不能弹系统目录选择器”，不是功能没做完，是环境限制
- 不要把 Web 限制误判成 Electron 缺陷

## 4. 本地 API 和预览服务要注意“旧进程”

最近已经反复出现过这种问题：
- 新代码加了新 intent
- 但预览页面连的还是旧的本地 API 进程
- 然后报：
  - `不支持的 intent: xxx`

排查顺序：
1. 先确认是不是旧 API 进程还活着
2. 再确认新代码有没有真正 build / restart
3. 再判断是不是逻辑问题

不要一看到 `INTENT_NOT_SUPPORTED` 就直接改业务代码。

---

## 4.1 先区分“真实 JS 错误”和浏览器静态资源噪音

最近已经再次遇到过这种情况：

- Web 预览里每页都显示 `Console: 1 errors`
- 实际只是浏览器默认请求 `/favicon.ico`
- 因为入口没显式声明 favicon，最终报了一个 `404 (Not Found)`

经验结论：

- 先看控制台原文，不要只看“有 1 个 error”就判断业务逻辑坏了
- 如果是：
  - `/favicon.ico` 404
  - React DevTools 开发提示
  这类通常不是业务故障
- 浏览器预览的 favicon 最稳做法是：
  - 在 `index.html` 里显式声明现有静态图标
  - 不要让浏览器自己去猜 `/favicon.ico`

## 5. 构建通过不代表运行时没问题

这个项目最近已经验证过：
- `npm run build` 通过
- 但运行时切页照样会炸

典型原因：
- 作用域错误
- 类型导入和运行时导入混用
- 只有切到某个页面才触发的运行时引用问题

所以每次改完都要做两层验证：
1. `npm run build`
2. 实际点页面、切入口、走一遍真实交互

不要只看 build。

## 6. 中文源码优先用 cat8 看

本机 PowerShell 已配置 `cat8`。

规则：
- 查看中文源码优先用 `cat8`
- 只要是 `tsx` 且文件里可能含中文 JSX / 中文文案 / 中文 tooltip，默认直接用 `cat8`
- 聊天 prompt、系统提示词、中文正则这类文件也默认直接用 `cat8`，不要把终端乱码误判成源码真的坏了
- 终端乱码不等于源码损坏
- 判断以这些为准：
  - 编辑器
  - `git diff`
  - `build`
  - 实际 app 行为

不要因为终端显示怪，就去“顺手修编码”，这很容易把大文件改坏。

## 7. 先判断“目标”，不要执着某一条命令

这条经验最早来自 `better-sqlite3`，但其实适用于很多问题。

思路：
- 目标是“把东西切到正确状态”
- 不是“必须让某一条命令成功”

例如：
- 目标是切换到 Web 版 / Electron 版 `better-sqlite3`
- 不是非要执着第一条 rebuild 命令一定成功

遇到问题时：
- 不要卡死在第一种方法上
- 在不扩大风险的前提下，优先尝试其他最小可行路径

## 8. 版本提交必须和中文说明同步

这条不展开细写，详细规则看：
- [D:\APP\finance-v1\PROJECT_MEMORY.md](/D:/APP/finance-v1/PROJECT_MEMORY.md)
- [D:\APP\finance-v1\CHANGELOG.zh-CN.md](/D:/APP/finance-v1/CHANGELOG.zh-CN.md)

这里只记最关键一条：
- 用户说 `git 0.0.x / git 下一个版本 / 这次版更 git`
- 默认不是普通 commit，而是“版本提交”
- 必须同步更新 `CHANGELOG.zh-CN.md`

补充硬规则：
- 版本提交做完后，不能停在 `git commit`
- 必须立刻补：
  - `git tag vX.Y.Z`
  - `git log --decorate -3`
  - `git tag --sort=-creatordate | Select-Object -First 5`
- 并同步更新这 4 个根文档的版本基线或最新状态：
  - `SYSTEM_ARCHITECTURE.md`
  - `CODING_PLAYBOOK.md`
  - `PROJECT_MEMORY.md`
  - `CURRENT_STATE.md`
- 只有确认最新提交上已经挂到对应 tag，版本提交才算真正收尾
- 如果只做了 commit 没打 tag，视为版本流程未完成，不要向用户汇报“版本已完成”

## 9. 数据库主键不要当成业务序号看

近期容易误判的点：
- `accounts.id`
- `categories.id`

结论：
- 它们是全表 `AUTOINCREMENT`
- 不是按 `book_id` 各自编号
- 多账本、删除、恢复、重建后，id 很大、跳号，都是正常现象

因此：
- 不要把主键不连续误判成数据坏了
- 如果页面想展示“从 1 开始”的顺序，要单独做显示序号，不要动主键设计

## 10. 模板、恢复、批量初始化都要用事务思维

凡是这种操作：
- 新建账本套模板
- 当前账本恢复
- 覆盖恢复
- 批量初始化分类和账户

必须优先用“整笔成功 / 整笔回滚”的思维去写。

## 11. 旧结构换新版面时，优先删旧 JSX，不要只靠 CSS 藏

这轮账户页改版已经再次验证：

- 只是在 CSS 里把旧工具栏、旧标题区 `display: none`
- 很容易被后面的旧样式重新覆盖
- 也容易让 JSX 里残留的旧文本节点继续渲染出重复内容

更稳的做法是：

- 新版结构确认生效后
- 直接把旧的 JSX 区块删掉
- 再补最小必要的样式

结论：
- 页面大改版时，`删旧结构 + 留新结构` 通常比“旧结构继续留着，靠样式压住”更稳。

不要接受这种实现：
- 先创建一半
- 失败了再留半套数据

这个项目已经明确采用：
- 单事务完成
- 任一步失败整笔回滚

## 12. 头像处理的经验

头像这块已经踩过真实坑。

结论：
- 头像不要长期存 data URL
- 正式模式应保存为本地 asset 文件
- PNG 经常带透明留白，不能直接原样当头像用

当前已验证更稳的做法：
1. 用户选图
2. 前端先裁掉透明留白
3. 再做正方形头像规格化
4. 后端把结果写到本地头像资源目录
5. 设置里只保留 `avatar_asset`
6. 返回给前端展示时，再派生 `avatar_data_url` 作为预览值

如果看到“头像下半截怪异”“透背景”“留大块空白”，优先查：
- 是否是 PNG 透明区
- 是否做了头像规范化
- 是否只是容器背景透出来
- 如果“头像已保存但预览裂掉 / 破图”，优先查：
  - 前端是不是直接拿本地 `avatar_asset` 路径渲染
  - 后端有没有返回 `avatar_data_url`
  - 前端预览是不是优先用了 `avatar_data_url`

## 16. 遇到滚动问题，先查“到底是谁在滚”

最近对话页已经踩出明确经验：

- 视觉上看到“整页在滚”，不代表应该直接上 `sticky`
- 如果外层容器本身在滚：
  - 内部 `sticky` 往往只是“粘在滚动容器里”
  - 不等于真的固定在页面上

排查顺序：
1. 先确认真正的滚动容器是谁
2. 再决定是：
   - 锁外层滚动
   - 还是让内层某个区域滚
3. 最后再决定要不要加 `sticky`

对话页当前已经验证更稳的结构是：
- 外层工作区不滚
- 左侧固定
- 右侧消息区内部滚
- 底部输入栏固定
- 如果还出现“对话卡没铺到底 / 输入栏悬空”：
  - 先补齐外层高度链路
  - 再让右侧主区改成 `flex` 纵向分配
  - 最后才考虑局部 `height` 或 `sticky`

## 17. 列表点击后“一闪一闪”，先查 effect 抢状态

最近对话页侧边栏已经踩出这类坑：

- 用户手动点了一次会话
- 页面里别的 `useEffect` 又根据：
  - 当前模式
  - 最近会话
  - 可见列表
  - 兜底逻辑
  再重设一次 `sessionId / selectedSkill`
- 结果就是：
  - 选中态闪一下
  - 列表像被“抢回去”
  - 容易误以为是 CSS hover / active 问题

经验规则：
- 先排查是不是多个 effect 同时在重置同一份选中状态
- 手动点击的优先级应高于自动兜底
- 自动选择逻辑只该在这些场景触发：
  - 首次进入页面
  - 当前项失效
  - 删除后兜底
  - 当前模式下确实没有选中项

不要一看到闪烁就先去改颜色、transition 或 hover。

## 18. 只要动了聊天 scope / 会话表 / 审计表，先想旧数据库迁移

这条已经连续踩过真实坑，必须单独记住。

近期出现过的典型问题包括：
- `scope CHECK` 还停留在旧枚举
- 旧库残留 `ai_sessions_old`
- 旧库残留 `ai_sessions_scope_old`
- 外键在 SQLite 重建过程中被自动改写到旧表名

结果表现通常是：
- 新增 `analysis / free_chat / entry` 后直接报错
- 看起来像业务逻辑坏了，实际上是旧数据库 schema 还没完全修干净

规则：
1. 只要改了这些东西：
   - `ai_sessions`
   - `ai_messages`
   - `chat_entry_drafts`
   - `ai_audit_log`
   - chat `scope`
2. 第一反应先检查：
   - 旧库有没有残留旧表名引用
   - `CHECK` 枚举有没有包含新 scope
   - 外键是不是还指向 `_old` 表
3. 不要只看“新建库 schema 正确”，还要验证“旧库迁移后也正确”

## 19. 修正若需要重启运行环境，结束时必须明确提醒

这条现在是项目级协作约定。

适用场景包括：
- 前端页面逻辑改动，但当前运行实例不会自动热更新到正式版
- Electron 主进程 / preload / IPC 改动
- 本地 API 进程改动
- 数据库自动迁移或自愈迁移

规则：
- 只要这次修正 **需要重启后才会真正生效**
- 结束时必须明确告诉用户：
  - 需要重启
  - 正式版怎么重启
  - Web 预览怎么重启
- 如果这次修正 **不需要重启**
  - 不必每次重复说明“无需重启”
  - 保持收尾信息简洁即可

推荐固定写法：
- 正式版：彻底关闭应用，再重新打开
- Web 预览：重启本地预览服务，再刷新页面

不要把这条埋在长说明里，要单独说清楚。

## 20. 对话页“自动切会话”逻辑要特别谨慎

最近已经踩出稳定经验：

- 新建对话草稿态里选模块，不应该自动跳到旧会话
- 只有“已有对话内主动切模块”，才允许跳到该类最近会话
- 发送消息、刷新列表、重算兜底时，不应再次抢走当前会话
- 手动点击某条会话的优先级，高于自动选择和最近会话兜底

如果出现这些现象：
- 刚发完消息就跳走
- 总被拉回置顶或最近会话
- 新建对话发出去后落到旧对话里

优先排查：
1. 是否把“草稿态切模块”和“已有会话切模块”混用了
2. 是否某个 `useEffect` 在列表刷新后又重设了 `sessionId`
3. 是否自动兜底逻辑覆盖了用户手动选择

## 21. 大文件里补排序 / 筛选联动时，别只接半截

最近流水页已经踩过一次直接运行时炸页的问题：

- 表头按钮和排序 state 已经接进去了
- 但真正的比较 helper 没补全
- 结果页面直接报：
  - `xxx is not defined`

经验规则：
- 在 `HomeLedgerPage.tsx` 这类大文件里补功能时，不要只改：
  - `useState`
  - JSX 按钮
  - 一半的排序分支
- 还要同步检查：
  - helper 函数是否已落地
  - `useMemo` 依赖是否补全
  - 分页重置依赖是否补全
  - 构建后页面是否真的能打开

尤其是“排序 / 搜索 / 筛选”这类横切逻辑，最容易出现“代码看上去只差一点，运行时却整页挂掉”的情况。

## 21.1 首页“最近流水”列宽没变化时，先查 grid 还是 flex

最近首页“最近流水”短日期列已经踩出一次很典型的错觉：

- 以为宽度是 `.overview-recent-date { flex-basis: ... }` 在控
- 实际桌面态真正生效的是 `.overview-recent-row { grid-template-columns: ... }`
- 结果改了 `flex` 那层，视觉上几乎没反应

经验规则：
- 先确认父容器当前到底是：
  - `display: grid`
  - 还是 `display: flex`
- 再去改真正控制列宽的那条规则
- 如果同一个选择器在文件里前后定义了两次，优先看后面的生效覆盖

## 22. 公共页头上的版本号不要手写

这轮已经把页头品牌区版本号收口成：

- 直接读取 `package.json.version`
- 由 `App.tsx` 公共层统一渲染

经验规则：
- 如果用户要做 `0.1.x` 这类版本提交
- 先改：
  - `package.json`
  - `package-lock.json`
- 再检查页头显示

不要再做这种实现：
- 在公共页头里写死 `v0.1.x`
- 每次发版再手改一遍 UI 文本

这样最容易出现：
- 版本 commit 已经变了
- 页面抬头却还是旧号
- 最后又多补一轮无意义改动

结论：
- 版本号属于包元数据
- 不是页面文案常量
- 以后优先保证“改包版本 -> 页头自动同步”

## 23. 下拉被截断时，先查 overflow 链，不要只调高度

这轮 `QuickEntryPage` 已经踩出非常明确的规律：

- 看到下拉底部被截掉
- 很容易误以为只是：
  - 高度不够
  - `max-height` 太小
  - 需要再缩一行

但真正根因往往是：
- 下拉仍然挂在原表单容器里
- 外层桌面壳层 / 工作区 / 内容卡片某一层带了 `overflow: hidden`
- 结果弹层再高也会被硬裁掉

更稳的处理顺序：
1. 先查它是不是仍然在原容器里 `absolute` 展开
2. 再查父层 overflow 链
3. 如果这是桌面工作区内的大浮层，优先改成：
   - `document.body` portal
   - `position: fixed`
   - 根据视口空间自动决定向上还是向下展开

不要先在 CSS 里反复试：
- `height`
- `max-height`
- `bottom`
- `transform`

这些通常只能缓解，不会真正治根。

## 24. Electron 里页面导出失败，先考虑主进程截图

这轮日历页再次验证：

- 页面一旦混入较复杂的：
  - 本地图标
  - 多层 SVG
  - `foreignObject`
  - 样式表里的 `url(...)`
- 前端自己拼 `svg -> canvas -> toDataURL()` 很容易报：
  - `tainted canvas may not be exported`

如果当前任务明确是桌面软件导出，不要默认死磕纯前端导出。

更稳的顺序是：
1. Renderer 算出目标区域矩形
2. 通过 `preload / IPC` 把区域传给 `main`
3. 用 `BrowserWindow.webContents.capturePage(...)` 直接截取窗口区域
4. 主进程写 PNG 文件

也就是说：
- 浏览器导出适合轻页面
- Electron 导出复杂页面时，优先用主进程原生截图

这通常比继续补 `data URL`、样式内联、资源白名单更稳。

## 25. 复用卡片边框阴影时，先查 CSS 变量作用域

这轮成员管理页已经踩出一个非常典型的误判：

- 看起来成员页和收支分类页用了同一套：
  - `border: 1px solid var(--ledger-card-border)`
  - `box-shadow: var(--ledger-card-shadow)`
- 但成员页最外层始终没有边框和阴影

真正根因不是：
- 选择器优先级
- 样式被覆盖
- 没热更新

而是：
- `--ledger-card-border`
- `--ledger-card-shadow`

这两个变量只在 `.ledger-page` 作用域里定义。

如果目标页面不在 `.ledger-page` 这套容器里，即使 CSS 写了：
- `var(--ledger-card-border)`
- `var(--ledger-card-shadow)`

实际也会取不到值，视觉上就像“样式完全没生效”。

以后遇到这种现象，排查顺序要改成：
1. 先查这两个问题：
   - 变量在哪里定义
   - 当前节点是不是在这个作用域里
2. 再查是不是：
   - 选择器优先级问题
   - 被其他规则覆盖
3. 如果页面不在变量作用域里：
   - 要么把变量提到更高层
   - 要么直接改成显式值

不要再先入为主地以为：
- “同样的 class 结构 = 同样会生效”

结论：
- 样式复用不只看类名，还要看变量作用域。

## 26. 修圆角与分割线衔接时，先分清“线在哪一层、补色在哪一层”

这轮成员管理页又踩出一个很典型的 UI 细节坑：

- 看到圆角和分割线之间露出一小块白色
- 很容易第一反应去：
  - 给整组加背景色
  - 给整行加背景色
  - 在单元格上直接补一块色

结果常见副作用是：
- 第二行或整组串色
- 弯曲分割线被截断
- 左右边角越修越脏

真正更稳的排查顺序应该是：
1. 先确认“哪条线”是谁画的
   - 是表头底线
   - 还是分组首行顶线
   - 还是外层壳的边框
2. 再确认“白缝”属于哪一层的背景露出来了
   - 是单元格层
   - 是表格层
   - 还是最外层壳背景
3. 最后再决定补色应该落在哪一层

这次成员管理页已经验证出更稳的经验：
- 不要直接给整组或整行染色去遮白缝
- 也不要优先在单元格伪元素上补色
- 如果白缝本质是外层壳背景露出来，优先在最外层壳上补一个很小的伪元素
- 同时让表格主体处在更高层，这样不会切断分割线
- 如果补丁色块是为了衔接“首行背景色”，那它本质上就是首行背景色的延伸：
  - 首行背景色一旦改色
  - 对应补丁色块颜色也必须同步更新
  - 不要把这类补丁色块写成和首行背景脱节的硬编码孤值

更稳的做法：
- 优先让补丁色块和首行背景共用同一个颜色变量
- 如果当前页面还没抽变量，至少在改首行背景时，把补丁色块当成同一组联动样式一起检查

一句话记忆：
- **线从哪一层画，补色就不要压到那一层上；白缝属于哪一层，就去那一层补。**

## 27. 动 MCP / Local API 生命周期时，先分清“是谁拉起来的”

这轮 XplorOne MCP 产品化已经踩出一个明确经验：

- Local API 可能是用户自己在设置里开启的
- 也可能只是 MCP 为了转发查询临时拉起来的

这两种来源语义不能混。

规则：
1. 如果 MCP 启动时临时拉起了 Local API
   - MCP 启动失败时，要把这次临时拉起的 API 一起回收
   - MCP 关闭时，也要把这次临时租用的 API 一起释放
2. 如果 Local API 本来就是用户主动开启的
   - MCP 不应该顺手把它关掉

结论：
- 以后只要动：
  - `localApiManager`
  - `localMcpManager`
  - `main.ts` 里的外部连接开关
- 第一反应先确认“当前这份 API 状态是谁拉起来的”
- 不要把“可运行”写成“谁都能关”的粗暴状态机

## 28. MCP token 只能在主进程里碰，renderer 只拿白名单桥

当前 XplorOne MCP token 已经切到：

- Electron `safeStorage`
- 本地加密元数据文件
- 主进程按需解密

这套边界不要再打破。

规则：
- `safeStorage` 只允许出现在主进程
- token 文件读写只允许主进程做
- renderer 不直接碰：
  - `safeStorage`
  - 文件系统
  - 原始 Electron API
- 设置页首屏只读脱敏元数据
- 只有这些动作才按需解密：
  - `显示完整`
  - `复制 token`
  - MCP Bearer 鉴权

如果后续改设置页里的 MCP 区域，优先保持：
- `Renderer -> preload bridge -> Main`
- 不要为了省事把 token 逻辑下放到页面层

## 29. 动技术身份前，先设计迁移，再改实现

这条现在是硬规则。

只要动这些东西：
- `baseDir`
- 数据目录名
- DB 文件名
- `appId`
- 环境变量前缀
- 凭据 namespace

第一反应都应该是：
1. 先判断旧数据怎么兼容
2. 先判断显式环境变量怎么兼容
3. 先判断迁移是否必须早于初始化
4. 先判断旧目录/旧文件是否只复制、不删除

不要直接做全局替换。

当前 XplorOne 技术身份迁移已经验证出的稳定原则是：
- `XPLORONE_*` 优先，`FINANCE_*` fallback
- 默认目录迁移采用复制，不自动删除旧目录
- 新旧 DB 文件名需要补齐兼容
- 主程序、独立 API、独立 MCP 必须共用同一套 runtime bootstrap

如果这类改动没先想迁移，就很容易出现：
- 新目录先被提前创建
- 新空库先被打开
- 旧数据看起来“像丢了”
- 某个 standalone 入口仍在走旧路径
